## 二、各类文档说明 + 脱敏要求

### A. BMS 产品/运维手册（PDF/Word → Markdown/PDF 脱敏版）

**用途**：RAG 回答「SOC/SOH、均衡、保护参数、通信异常」等。

**建议章节（有则入库，无则跳过）**：
- 系统概述、电气参数、保护策略
- 告警列表与处理建议
- 通信接口（CAN/Modbus/485 等）说明
- 常见故障排查流程

**脱敏清单**：
| 原始字段 | 处理方式 |
|----------|----------|
| 客户名称、电站名、项目编号 | 替换为 `客户A`、`站点001` |
| 联系人、电话、邮箱 | 删除或 `138****0000` |
| 精确经纬度、详细地址 | 改为「某省某市」 |
| 内部未公开参数/密钥 | 删除整段 |
| 公司机密报价、合同条款 | 不纳入 |

**文件命名**：`doc_bms_manual_v1.pdf`（或 `.md`）

---

### B. PCS/逆变器运维手册

**用途**：补充交流侧故障（并网、谐波、过温、直流侧异常等）。

**脱敏**：同 A。

**文件命名**：`doc_pcs_manual_v1.pdf`

---

### C. 故障码对照表（结构化，强烈建议 Excel/CSV）

**用途**：
- 第2月：RAG 可检索「告警码含义」
- 第3月：Agent **Tool 1** 直接查表（比 RAG 更准）

**建议字段**：

```csv
alarm_code,alarm_name,level,subsystem,description,suggested_action,reference_page
E101,单体过压,严重,BMS,单体电压超过阈值,检查均衡与充电策略,12
W205,通信中断,警告,EMS,与BMS通信超时,检查链路与波特率,45
```

**最少行数**：30 条（覆盖 BMS/PCS/EMS/环境各若干条）  
**理想行数**：80–150 条

**脱敏**：无客户信息即可；码表本身通常是通用技术资料。

**文件命名**：
- `fault_codes.csv`（Agent 工具用）
- `fault_codes.md`（RAG 可读版，可由 CSV 导出）

---

### D. 历史故障工单摘要（脱敏，第3月核心）

**用途**：Agent 学习「输入告警 → 分类 → 根因 → 处置步骤」；RAG 学习案例表述。

**每条工单建议结构（JSON 或 Markdown）**：

```json
{
  "ticket_id": "TK2024-0001",
  "site_id": "SITE001",
  "device_type": "BMS",
  "alarm_codes": ["E101", "W205"],
  "symptom": "充电末期频繁过压告警，伴随通信偶发中断",
  "root_cause": "单体一致性偏差+通信线接触不良",
  "actions": ["停机检查均衡配置", "紧固通信线缆", "复测24h"],
  "resolved": true,
  "duration_hours": 6
}
```

**数量**：
- 最少：20 条（每类故障至少 2 条）
- 推荐：50 条

**脱敏清单**：
| 字段 | 处理 |
|------|------|
| 真实工单号 | 改为 `TK2024-0001` 连续编号 |
| 人名 | `工程师A` |
| 客户/电站 | `SITE001` |
| 具体日期 | 可保留年月，日改为 `01` |
| 内部追责、金额 | 删除 |

**文件命名**：`tickets_desensitized.jsonl` 或 `tickets_desensitized.md`

---

### E. 储能 FAQ（运维问答）

**用途**：RAG 评估集 + 高频问题缓存。

**格式**：Q/A 对，每条 50–300 字。

**示例**：
- Q：SOC 和 SOH 区别？  
- Q：通信中断先查什么？  
- Q：过温告警能否自动恢复？

**数量**：10–30 条（可自行从手册提炼，不必等公司文档）

**文件命名**：`faq_energy_storage.md`

### F. EMS/协议说明（摘要即可）

**用途**：第3月 Mock「设备上报」、Agent 上下文（不必真联硬件）。

**准备内容**（1–3 页 Markdown）：
- 支持的寄存器/告警上报格式（示例）
- 典型告警 JSON Mock 样例 5 条

**文件命名**：`ems_protocol_summary.md`、`mock_alarms.json`

---

### G. 故障 Taxonomy 分类表（建议自建，P0）

**用途**：Agent 分支路由（过充 / 欠温 / 通信 / 绝缘 / SOC 异常等）。

| 一级分类 | 二级示例 | 关联告警码示例 |
|----------|----------|----------------|
| 电池安全 | 过压/欠压/过温 | E101, E102 |
| 通信 | BMS/EMS/PCS 中断 | W205, W210 |
| 功率转换 | PCS 故障、并网异常 | E301 |
| 环境 | 舱温、湿度、消防 | W401 |

**文件命名**：`fault_taxonomy.md`

---

### H. 处置 SOP（标准作业流程）

**用途**：Agent 输出「处置步骤」的模板依据；避免模型胡编。

**每条 SOP 包含**：
- 适用告警类型
- 前置安全检查（停电/验电/挂牌）
- 步骤 1–N（可操作）
- 验收标准
- 升级条件（何时转人工）

**数量**：5–10 条（与 taxonomy 一级分类对应）

**文件命名**：`sop_desensitized.md`

## 六、脱敏工作流（在职场景可执行）

1. **申请**：向公司申请「技术资料用于个人学习/脱敏副本」，书面或邮件留痕。  
2. **导出**：只导出必要章节（PDF 打印为 PDF 再处理，或 Word 另存）。  
3. **脱敏**：用查找替换 + 人工抽检（客户名、站点、人员）。  
4. **审查**：自己过一遍 + 请同事扫一眼（可选）。  
5. **存储**：仅放 `~/energy-ai-data/desensitized/`，Git 仓库只放 `.gitignore` 规则。  
6. **演示**：对外 Demo 只用 `SITE001`、合成告警，不用真实电站截图。

**`.gitignore` 建议追加**：
```gitignore
energy-ai-data/
*.raw.pdf
tickets_raw.*
```
